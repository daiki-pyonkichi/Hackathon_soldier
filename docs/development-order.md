# 開発の進め方とロードマップ

> ゴール: 「いつ何を作る」「いつ他の人とすり合わせる」がわかる。

## 💡 作る順番の原則

1. **型を決める**（データの形を決める）
2. **ダミーデータで画面を作る**
3. **APIを他人と合わせる**（URLとリクエスト/レスポンスの形を文字で決める）
4. **サーバーを作る**
5. **画面とサーバーをつなぐ**
6. **さらに仕上げる**（エラー表示、ローディング、デザイン）

🚫 **アンチパターン**: デザインを先に完成させようとしない。**まず動くものを作る**、見た目は最後に仕上げる。

## 🤝 すり合わせのコツ

### 毎日（夜20時頃を目安に）

- LINEで一言進捗報告。「今日はここまでやった」という一行でOK
- 詰まってるときは「ここで詰まってる」と言う
- 長縄はニュースを見てヘルプに入る

### 週に1回のポイントで集まる（できれば）

- 5/24頃: データモデルとAPI設計をすり合わせ
- 5/27頃: 中間チェック。ログインと在室一覧がつながるかを見る
- 6/2: 研究室で実機テスト

## 🔗 誰と何をすり合わせるか（依存関係）

```
kuremoto(キャラ素材) ─PNGファイル─→ takebayashi(画面)
naganawa(ログイン) ─user型・token仕様─→ tsutsumi, takebayashi
tsutsumi(presence API) ─URLとレスポンス形式─→ takebayashi
kuremoto(見取り図モック) ─レイアウト案─→ takebayashi
```

### 重要な「接面」ポイント

1. **User型の定義**: naganawa と tsutsumi で合わせる（userに何の項目があるか）
2. **tokenの形式**: naganawa が決めて takebayashi に伝える
3. **APIのURLとレスポンス**: tsutsumi が決めて takebayashi に伝える
4. **キャラと状態の対応**: kuremoto が決めて takebayashi に伝える

## 🚦 「ここまで来たら合流」チェックポイント

以下のタイミングで **一旦作業を終えて他の人とチェック**。

| タイミング | チェック内容 |
| --- | --- |
| セットアップ完了 | 全員がローカルでアプリを起動できたか |
| 認証設計完了 (naganawa) | tsutsumiとuserテーブル設計を合わせる |
| DB設計完了 (tsutsumi) | naganawaとuserテーブルを合わせる |
| APIドラフト (tsutsumi) | takebayashiにURL/入出力形式を伝える |
| ログインAPI実装完了 (naganawa) | takebayashiにtoken仕様を伝える |

| ログイン画面動作 | 3人で動作チェック。トークン保存、リロードしてもログイン状態維持 |
| ホーム画面表示 | ログイン後にホームへ、pingが送信されてるか |
| 研究室テスト前 | **全員**でIP判定の動作をチェック |

## 📢 コミュニケーションのルール

- **詰まったら 30分以内に言う**。ソロでハマるのもったいない
- コードの質問は **スクショと何をしたいか** をセットで送る
- PR にレビューコメントをつけるとコードの記録が残るので おすすめ

## ✅ このページで言いたいこと

- 作らないと進まない。まず手を動かす
- ひとりで詰まらない。スケジュールより人を頼る
- **個々の作業 → 接面ですり合わせ → 結合 → テスト** のサイクル
